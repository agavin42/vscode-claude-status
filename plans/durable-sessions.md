# Durable Sessions

## Conceptual

Today a "Claude session" in this extension is the `vscode.Terminal` object. The `TrackedTerminal` record exists only as long as that terminal exists; close the terminal and the record dies with it. `workspaceState` persists a thin shadow (ccId + customName + displayName + order) so we can re-bind to a terminal on reload, but the record itself is not the unit of identity — the terminal is.

This plan inverts that. A **Session** becomes a durable, JSON-serializable record that *owns* a terminal as one of its transient runtime fields. The session can outlive the terminal, the VS Code window, and the Claude Code process; it knows its directory, its Claude session ID, its name, and how to recreate its terminal from cold. Killing or losing the terminal degrades the session to a "cold" state, not nonexistence.

This unlocks four user-facing capabilities:

1. **Remake** — kill the current terminal and recreate it in the same directory with `claude --resume <claudeSessionId>`, faithfully picking up the conversation. Useful for picking up a new Claude version mid-work, and for the "Claude got confused, reboot it" reset case.
2. **Survive VS Code restarts** — sessions persist as cold entries in the panel; clicking remake brings them back with the right resume.
3. **Real settings dialog** — a webview-based form for editing name and directory together (plus read-only context: ccId, claudeSessionId, version, created-at), instead of the current single-field input box.
4. **Richer tooltips** — name, directory, Claude session ID, version, state, last prompt.

The throughline: **the record is the truth, the terminal is an attachment**.

## Design

### Session record

```ts
// Persistent — serialized to workspaceState
interface PersistedSession {
  id: string;                    // ccId (cc-{ts}-{rand}), stable for lifetime of session
  displayName: string;           // "s:HH:MM:SS" generated at creation
  customName?: string;           // user-assigned via settings dialog
  directory: string;             // absolute cwd; survives remake. Updated from hook .cwd field
  claudeSessionId: string;       // Claude Code's session id (UUID). Pre-generated by extension and
                                 // passed via --session-id at create. Updated from hook on every event
                                 // (in case it changes via fork or compact).
  claudeVersion?: string;        // "claude --version" output, refreshed on remake
  parentSessionId?: string;      // source session's claudeSessionId if this was forked
  parentDisplayName?: string;    // human-readable lineage hint for tooltip / webview
  createdAt: number;             // epoch ms
}

// Transient — in-memory only, rebuilt on activate
interface RuntimeSession extends PersistedSession {
  terminal?: vscode.Terminal;    // undefined ⇒ cold
  state: ClaudeState;            // current live state
  lastPrompt?: string;
  permsEnteredAt?: number;
  pollInterval?: NodeJS.Timeout;
  hasUserInput: boolean;
  transcriptPath?: string;       // from hook .transcript_path (transient — re-derived each run)
  subagentCount: number;         // running subagents (from SubagentStart/SubagentStop events)
}

type Session = RuntimeSession;
```

`claudeSessionId` is now non-optional (not `?:`) — we pre-generate the UUID at terminal creation and pass it via `--session-id <uuid>`. The hook keeps refreshing it on every event, so fork or compact-time id changes are followed automatically. Cold sessions still have a valid `claudeSessionId` because creation runs before any cold→warm transitions could happen.

`terminal === undefined` is the **cold** indicator. Everything else is either runtime housekeeping or already persisted.

### SessionStore

A single class owns the `Map<id, Session>`, the order list, save/load, and lifecycle transitions. Replaces the current top-level `trackedTerminals` map, `terminalOrder` array, and `saveState` / `restoreState` free functions. Lives in `src/extension.ts` for now; can be split into its own file if it grows past ~200 lines.

Surface:

```ts
class SessionStore {
  create(args: { resume?: boolean; dir?: string; forkFrom?: string }): Session;
  adopt(terminal: vscode.Terminal): Session;
  remake(id: string): Promise<void>;          // kills terminal, recreates in same dir w/ --resume
  suspend(id: string): void;                  // kills terminal, leaves session as cold
  delete(id: string): void;                   // drops session entirely + state files (confirmation lives at command layer)
  update(id: string, patch: Partial<PersistedSession>): void;
  bindTerminal(id: string, t: vscode.Terminal): void;
  get(id: string): Session | undefined;
  all(): Session[];                            // in custom order
  reorder(ids: string[]): void;
  save(): void;                                // serializes sessions + order atomically
  restore(): void;                             // restores both, then save()s to flush self-heal
}
```

### Order persistence (fix for existing "sometimes resets" bug)

The current implementation has a real bug: `terminalOrder` is mutated from `getChildren()` (a read path) without calling `saveState()`. Combined with `createClaudeTerminal` not pushing to the order before saving, the sequence `[create new terminal → tree refresh → reload window]` loses the new terminal's persisted position. Drag-drop saves correctly because `handleDrop` calls `saveState`; the failure mode is creation-without-subsequent-interaction.

`SessionStore` fixes this by making order a first-class field with these invariants:

- **Every mutation calls `save()`.** No read path mutates state. `create`, `adopt`, `delete`, `reorder`, `remake`, `bindTerminal`, `update` all save before returning.
- **Order is part of the same atomic write as sessions.** `save()` serializes both in one `workspaceState.update` cycle, so you can't get a half-saved state where sessions and order disagree.
- **Order references session ids, not terminal-bound things.** Cold sessions stay in their slot across terminal disposal and remake. Filtering on "is this terminal alive" is wrong under the new model — `terminalOrder.filter(id => trackedTerminals.has(id))` in the current `restoreState` would drop every cold entry.
- **`restore()` calls `save()` at the end.** If terminal-name matching only catches some persisted ids (e.g. 5-retry window expires), the in-memory state and disk state diverge today. The new code flushes the reconciled state immediately, so the disk always reflects what's actually loaded.
- **`getChildren()` is pure** — calls `store.all()`, returns the array as-is. Any "new session not in order" reconciliation happens inside the store at mutation time, never on read.

A close affordance currently means "delete the session entirely." Under the new model we have two distinct operations with distinct names:

- **Suspend** — terminal dies, session enters cold state. Inline icon (the existing trash slot, relabeled). Most common case; cheap and reversible (Remake brings it back).
- **Delete** — kills terminal AND drops the session record + state files. Right-click only, no inline icon, with a confirmation prompt. Destructive, less frequent.

The naming is deliberate: "close" is ambiguous between the two, so we don't use it. "Suspend" reads as reversible; "Delete" reads as final.

### Lifecycle states

```
   ┌─────────┐  create        ┌────────────┐
   │  void   ├───────────────►│   warm     │◄────┐
   └─────────┘                └─────┬──────┘     │ remake
                                    │ suspend    │
                                    ▼            │
                              ┌────────────┐     │
                              │   cold     ├─────┘
                              └─────┬──────┘
                                    │ delete (warm or cold)
                                    ▼
                                ┌────────┐
                                │  void  │
                                └────────┘
```

`warm` = terminal alive, polling. `cold` = persisted record, no terminal, no polling. Both render in the tree view; cold sessions use a snowflake icon and omit live state badges. `delete` is reachable from either state and is the only path back to void.

### Hook contract additions

Verified against the Claude Code docs (https://code.claude.com/docs/en/hooks, https://code.claude.com/docs/en/cli-reference) on 2026-05-24.

**Always-present input fields on every event:** `session_id`, `transcript_path`, `cwd`, `hook_event_name`. The current hook only extracts `hook_event_name`, `tool_name`, `prompt`, `notification`. We can pull the other three trivially.

**Capture from every event** (idempotent overwrites — values are stable mid-session except where noted):

| File             | Source                              | Notes                                                                                                            |
| ---------------- | ----------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `{ccId}.session` | `.session_id`                       | Stable; may change across compaction — see `PostCompact` below                                                  |
| `{ccId}.cwd`     | `.cwd`                              | Updates as Claude changes directory; `CwdChanged` event makes it explicit                                       |
| `{ccId}.tx`      | `.transcript_path`                  | Path to the JSONL transcript file; tracks the live transcript even if a fork or compact rewrites it             |
| `{ccId}.version` | `claude --version` (backgrounded)   | Captured once at `SessionStart` only, gated on file existence. Backgrounded subshell so it doesn't slow the hook |

Reading version from inside the hook is more robust than reading from extension side: it captures the actual `claude` that the user's shell resolved (PATH, nvm shims, custom wrappers), not whatever `${claudeCodeStatus.command}` resolves to from the extension host.

**Hook events we currently subscribe to:** `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop`, `UserPromptSubmit`, `Notification` (matchers), `SessionStart`, `SessionEnd`.

**Hook events to add:**

| Event             | Why                                                                                                                                                                   |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CwdChanged`      | Update `{ccId}.cwd` so `session.directory` follows Claude through `cd`. Without this, Remake recreates in the original dir even if the user has moved.               |
| `PreCompact`      | Optional state "compacting" for the UI. Also a chokepoint for refreshing `{ccId}.session` since session_id may change here.                                          |
| `PostCompact`     | Recapture `session_id`. Resolves the open question about session-id mid-session changes — we pin to the latest known value rather than the first.                  |
| `PermissionDenied`| Cleanly transition out of PERMS when the user denied (instead of relying on the timeout). State → IDLE.                                                              |
| `StopFailure`     | Treat as IDLE (safe default — session isn't actively working).                                                                                                       |

**Notification matchers are wrong in the current install script.** The current config uses `idle_timeout` and `waiting_for_user_action`. The actual matchers per docs are:

- `permission_prompt` — Claude is asking for permission (overlaps with `PermissionRequest` event; redundant but harmless to subscribe to both)
- `idle_prompt` — replacement for what we were calling `idle_timeout`
- `auth_success`
- `elicitation_dialog` / `elicitation_complete` / `elicitation_response`

Plan updates `install-hooks.sh` to use the correct matchers (`permission_prompt` and `idle_prompt`), and the hook script's `Notification)` branch likewise. Existing Andy-machine `~/.claude/settings.json` will need re-running the install script to pick up the corrected matchers — install script is idempotent by design (jq-merges into existing settings) so this is safe.

### Remake mechanics

```
remake(id):
  s = sessions.get(id)
  if s.terminal: s.terminal.dispose()             // triggers onDidCloseTerminal, but we suppress delete (see below)
  reset {ccId}.state → IDLE
  unlink {ccId}.prompt and {ccId}.version (stale)
  newTerminal = vscode.window.createTerminal({
    name: `CC: idle [${s.displayName}]`,
    cwd: s.directory,
    env: { VSCODE_CC_ID: s.id },                  // same ccId — state files stay aligned
  })
  cmd = `${config.command} --resume ${s.claudeSessionId} ${extraArgs.join(' ')}`
  if config.remoteControl: cmd += ` --remote-control -n "${s.displayName}"`
  newTerminal.sendText(cmd)
  s.terminal = newTerminal
  s.state = Idle
  s.hasUserInput = false                          // re-arm startup-BUSY suppression
  s.lastPrompt = undefined
  watchStateFile(s)
  save()
```

For **create** (cold start, new session), the launch command is:

```
${command} --session-id <pre-generated-uuid> ${extraArgs} [--remote-control -n "${displayName}"]
```

The extension generates the UUID via `crypto.randomUUID()` before creating the terminal and stores it as `session.claudeSessionId` immediately. No race window where the id is unknown.

For **fork**, we cannot use `--session-id` (Claude generates a new id under `--fork-session` semantics, and explicitly setting one is ambiguous). The launch command is:

```
${command} --resume ${src.claudeSessionId} --fork-session ${extraArgs}
```

The new session's `claudeSessionId` is initially set to a placeholder UUID and overwritten by the hook on the first event. This is the one narrow race window left in the design; if it matters, we can fall back to running the new claude with `--print --max-turns 0 --resume <src> --fork-session` to provoke the hook before the user types — but probably not needed.

**Suppressing the delete-on-dispose:** the existing `onDidCloseTerminal` handler deletes the tracked terminal entry. Under the new model, terminal disposal ≠ session deletion. Two ways to handle this:

- (a) Set a transient `s.remakeInFlight = true` flag before disposing, check in the handler, skip delete if true.
- (b) Change the handler to "mark cold" instead of "delete," and add an explicit `delete` user action elsewhere.

(b) is the cleaner model — see "Lifecycle states" above. The handler now does: stop polling, clear `terminal`, leave the session record in place. Delete happens only via the explicit menu item.

### Restore on activate

Current: try to match restored terminals to persisted ccIds by short-id substring in `terminal.name`; persisted entries with no matching terminal are silently dropped.

New: every persisted session is restored as a `Session` record regardless. Then:

- For each `vscode.window.terminals` entry whose name contains a known ccId's `displayName` (e.g. `s:14:32:07`), bind it → warm.
- Persisted sessions with no matching terminal stay cold.
- The existing 5×500ms retry pattern is preserved (terminal names take a moment to populate post-reload).

The state-file-reset-to-IDLE behavior moves into the warm-bind path; cold sessions don't touch state files until remake.

### Status bar updates

Current: `CC: idle, BUSY(2)` with errorBackground when any PERMS exists.

New format includes a cold count (only when > 0): `CC: idle, BUSY(2), ❄️(3)`. Cold sessions don't contribute to the urgency-color logic — only warm warnings (BUSY, PERMS, WAITING) and timeouts affect the bar color.

### Subagent surfacing

Hook subscribes to `SubagentStart` and `SubagentStop`. Each event increments / decrements a counter file `{ccId}.subagents` (atomic via a small lock or per-event fresh write of the integer). Extension reads it in the poll cycle and surfaces:

- **Tooltip:** "🟡 BUSY · N subagents" when count > 0
- **Tree row description:** optional small badge `[N]` after the state prefix when count > 0
- **Status bar:** not surfaced in v1 (state-only aggregation stays simple)

This is the minimum subagent visibility for v1. Broader subagent support (per-subagent state, drill-down, task-list integration) is a follow-up plan beyond this scope.

### Sibling actions: heal, sibling, fork

Three actions that complement the lifecycle commands. All act on the right-clicked session as the source. Sibling and Fork both create a new session related to the source; the difference is whether they inherit the conversation context.

|              | Sibling                   | Fork                                                  |
| ------------ | ------------------------- | ----------------------------------------------------- |
| Directory    | Same as source            | Same as source                                        |
| Conversation | Cold start (fresh Claude) | Resumes source's conversation at fork moment          |
| Lineage      | Independent               | `parentSessionId` recorded                            |
| Use case     | "Different task, same dir" | "Branch off this conversation without disturbing it" |

**Reconnect remote-control.** When `--remote-control` drops (Claude Code app disconnects, network blips), the manual fix is typing `/remote-control` in the running session. This action does that programmatically:

```ts
reconnectRemoteControl(id):
  s = sessions.get(id)
  if !s.terminal: return                              // cold, no-op
  s.terminal.sendText("/remote-control")
```

No detection — the action is user-triggered when they notice the app isn't seeing the session. Menu item is gated on a context key `claudeCodeStatus.remoteControlEnabled` set from the config value at activation and refreshed via `onDidChangeConfiguration`. Without that gate, the item shows for users who don't have remote-control on, which is confusing.

**New sibling.** Create a new cold-start session with `cwd = source.directory`. If user provides a name, it follows the existing rename flow (which already issues `/rename` when remote-control is on).

```ts
newSibling(sourceId):
  src = sessions.get(sourceId)
  name = await showInputBox({prompt: "Name (optional)"})
  newSession = store.create({ dir: src.directory })
  if name: store.update(newSession.id, { customName: name })
```

This is just `createClaudeTerminal` with an explicit `cwd` override and an optional name pre-fill — most of the implementation already exists. The only new wiring is the menu entry + override pathway through `SessionStore.create`.

**Fork.** Create a new session that inherits the source's conversation up to the fork moment, then diverges. Equivalent to "branch this conversation."

Claude Code has a native `--fork-session` flag (verified in the CLI reference): "When resuming, create a new session ID instead of reusing the original (use with `--resume` or `--continue`)". So fork is just:

```
claude --resume <source_session_id> --fork-session
```

`claude` generates a new session id internally for the forked session; we capture it via the hook's `session_id` field on the first event in the new terminal, exactly the same path as a normal new session. No transcript-file copying, no project-slug derivation, no race window.

```ts
fork(sourceId, name?):
  src = sessions.get(sourceId)
  if !src.claudeSessionId: throw "source has no Claude session id yet — wait for first interaction"
  if src.state == Busy: confirm "Source is busy — fork now anyway?"
  newSession = store.create({
    dir: src.directory,
    forkFrom: src.claudeSessionId,                  // recorded as parentSessionId once new id is captured
    parentDisplayName: src.customName ?? src.displayName,
  })
  // store.create launches terminal with: claude --resume <src.claudeSessionId> --fork-session
  // The new claudeSessionId is captured by the hook on first event in the new terminal.
  if name: store.update(newSession.id, { customName: name })
```

`SessionStore.create` gains an optional `forkFrom: string` parameter that drives the `--resume <id> --fork-session` flags on the launch command line.

**Race caveat (much milder than before):** if source is mid-conversation when the fork is launched, the forked session resumes from whatever state is on disk at that moment — same semantics as `--resume` against an active session. We still warn if source is BUSY: "Source is active — fork now anyway?"

### Phase 8 — Callback channel: Claude-driven session control

Lets Claude (running inside a session via `--remote-control`) reach back into the extension to spawn, heal, remake, rename, or list sessions. The lever is a small CLI (`cc-status`) that Claude invokes via Bash; the wire is a file-based RPC consistent with the existing state-file pattern.

#### Channel design

A new sub-dir alongside the state dir:

```
$TMPDIR/claude-code-status/
  ├── {ccId}.state             ← existing
  ├── {ccId}.prompt            ← existing
  ├── {ccId}.session           ← Phase 1
  ├── {ccId}.version           ← Phase 1
  └── cmd/
       ├── {nonce}.req         ← CLI writes JSON request
       ├── {nonce}.taken       ← extension atomically renames before processing
       └── {nonce}.res         ← extension writes response; CLI optionally reads
```

Request JSON shape:

```json
{
  "id": "nonce-uuid",
  "from": "cc-1737293849234-a3f9b1",
  "cmd": "spawn",
  "args": { "name": "debug-tunnel", "dir": "/Users/.../repo" }
}
```

Response JSON:

```json
{ "ok": true, "result": { "id": "cc-1737293901123-c8e2d4" } }
```

or:

```json
{ "ok": false, "error": "session 'debug-tunnel' not found" }
```

**Atomic claim:** extension polls `cmd/*.req`, renames each to `.taken` before reading (atomic on Linux/macOS via `rename(2)`). The instance that wins the rename owns the command. Other instances skip. This prevents double-execution across multi-window setups on the same workspace.

**Multi-window targeting:** when a command targets a specific ccId, only the extension instance that has that ccId in its `SessionStore` should execute. If the rename winner doesn't own the target, it writes `{nonce}.res` with `{ ok: false, error: "not owned by this window" }` and other instances are too late — so the CLI sees an error and the user knows to try from the right window. Alternative for v1: cross-window commands are out-of-scope; the CLI just reports "session not found" if no instance answers within a timeout.

**Polling cadence:** the existing 100ms state-file poll already runs. Add the `cmd/` scan to the same tick (cheap — usually empty dir).

#### CLI surface (`cc-status`)

```
cc-status sibling [--name NAME] [--dir DIR]              # cold-start new session; defaults dir = $PWD
cc-status fork    [--name NAME] [--id ID | --name NAME-OF-SOURCE]
                                                         # fork from source (default: self via $VSCODE_CC_ID)
cc-status heal    [--id ID | --name NAME]                # remote-control reconnect
cc-status remake  [--id ID | --name NAME]                # defaults to self via $VSCODE_CC_ID
cc-status rename  --name NAME [--id ID]                  # defaults to self
cc-status list    [--format json|tsv]                    # all sessions, including cold
cc-status get     [--id ID | --name NAME]                # one session's full record
```

`fork` without a source is the most useful Claude-side shape: "fork me into a new sibling." `fork --name "review-branch"` from inside a working session creates a new session that inherits the conversation. The source keeps going untouched.

- Self-targeting via `$VSCODE_CC_ID` env (no `--id` needed for self-actions like `remake`, `rename`).
- Sibling targeting via either `--id` (exact) or `--name` (custom or display name).
- All commands wait up to 2s (configurable) for a `.res` file, then exit.
- `spawn` without `--dir` defaults to the caller's `$PWD`, NOT the source session's directory — Claude is invoking from a shell that has its own cwd, which is usually the right thing.

#### CLI implementation

Shell or small Node script. Shell is simpler:
- Writes `$TMPDIR/claude-code-status/cmd/{uuid}.req` with the JSON
- Polls for `{uuid}.res` up to 2s
- Prints result, exits with `0` if `ok`, `1` otherwise

Lives in `~/bin/cc-status` (alongside the existing `cc-status-hook.sh`). Installed by `install-hooks.sh`.

#### Discoverability to Claude

Two pieces, both installed by `install-hooks.sh`:

**1. `~/.claude/CLAUDE.md` fragment** — appended idempotently (looks for a marker comment before inserting). Tells Claude what `cc-status` is, when to use it (spawning subtask sessions, healing siblings, forking conversation context), and how the slash commands map onto it.

**2. Single slash command `/cc`** at `~/.claude/commands/cc.md` — one umbrella command with subcommands matching the CLI. Avoids cluttering the slash namespace with seven near-identical entries.

```markdown
---
description: Control Claude session panel (sibling, fork, heal, remake, rename, list, get)
argument-hint: <subcommand> [args]
---

<!-- managed by claude-code-status install-hooks.sh -->

Run: `cc-status $ARGUMENTS`

Then briefly report what happened — name and id of the new / changed session, or the error if it failed.
```

Usage from inside Claude:

```
/cc fork debug-branch
/cc sibling new-task
/cc heal main-thread
/cc remake
/cc rename my-feature
/cc list
/cc get debug-branch
```

`$ARGUMENTS` passes through everything after `/cc`, so the slash command is a thin pass-through. All subcommand parsing and validation lives in the `cc-status` CLI — no duplication.

**Install / update behavior:**

- Install script `install-hooks.sh` creates or overwrites `~/.claude/commands/cc.md` on every run. Idempotent — re-running picks up newer command bodies if we change them later. The marker comment at the top tells humans not to edit by hand.
- `--uninstall` removes `cc.md`.
- `--verify` checks the file exists and contains the marker.

This way `/cc fork debug-branch` inside any Claude session yields the same result as `cc-status fork --name debug-branch` in Bash, with `/cc<TAB>` showing the argument-hint for discoverability.

### Open questions (resolved)

All eight resolved in plan review — recorded here for traceability.

1. **Close vs delete UX** → **Two actions: Suspend and Delete.**
   - **Suspend** — terminal dies, session record stays as cold. Inline icon (was the "close" trash). Most common case.
   - **Delete** — terminal dies AND session record dropped, state files unlinked. Right-click only (no inline icon), with confirmation prompt. Destructive, less frequent.
2. **Cold-session auto-expire** → **Leave to user.** No background pruning. Delete is a user action.
3. **Status bar cold sessions** → **Separate count.** Format: `CC: idle, BUSY(2), ❄️(3)`. Cold sessions don't contribute to the urgency-color logic (PERMS-red, BUSY-yellow).
4. **Multi-window** → **First-window-wins, second sees cold.** Andy doesn't use multi-window; pick stands.
5. ~~**claudeSessionId mid-session changes**~~ → resolved by docs. We re-extract `session_id` on every hook event, so the latest known id is always live.
6. **Sibling on cold sessions** → **Looser version: allow it.** Directory is on the persisted record, so cold sources work.
7. **Subagent activity in tooltip** → **Include in v1.** Hook subscribes to `SubagentStart`/`SubagentStop`, maintains a per-ccId counter in `{ccId}.subagents`, extension reads and surfaces in tooltip and (optionally) tree label. Broader subagent support is a follow-up plan after this scope ships.
8. **`--session-id <UUID>` shortcut** → **Use it.** Extension generates the UUID at terminal creation (Node's `crypto.randomUUID()`), passes `--session-id <uuid>`, and stores it as `session.claudeSessionId` upfront. Eliminates the "wait for first hook event" race for cold starts. (Fork still relies on hook capture because `--fork-session` issues its own new id — but that's a narrower window.)
